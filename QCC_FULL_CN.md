# 第一章　QCC 是什么

---

## 1.1　QCC 的本质定义

QCC 是一个面向**可信账本与可信数据**的区块链操作系统。

它以**字节数据可信上链**为基础、以**规则驱动执行**为核心、以**资产与数据统一流通**为目标，同时承担两种角色：作为**价值网络**，处理资产的发行、流转与归属；作为**数据网络**，处理真实世界信息的可信记录、结构化发布与长期存证。

QCC 彻底重构了价值与数据的关系：它不以资产投机为终点（虽然也可以投机），而以**支撑真实世界的复杂业务逻辑**为起点。

> QCC 不是一条转账型公链，不是单一存证链，也不是普通智能合约平台。QCC 的目标不是由官方完成所有应用，而是为第三方开发者、企业与机构提供底层链上运行环境。

---

## 1.2　QCC 的第一性原理

区块链的第一性原理不是发币，而是**不可篡改**。

不可篡改的本质价值，不应只服务于转账记录。它应该进一步延伸到：

- **可信数据**：让任意字节数据具备不可篡改的可信来源
- **可信账本**：让资产、权属、规则的变更有迹可查
- **可验证行为**：让链上操作具备明确语义和可验证后果
- **可审计规则**：让业务规则公开、透明、可确认
- **可追溯状态**：让任意时刻的系统状态都可回溯

QCC 正是沿着这条第一性原理向前推进——把不可篡改从「转账账本」扩展到「可信字节网络」。

## 1.4　核心概念对照

理解了 iOS 的系统逻辑，QCC 的每一个核心设计决策都能找到对应的直觉：

**容器 = 进程**　开发者可申请、可拥有、可扩展的原生链上运行单元。每个容器有独立业务边界，参与系统的验证、执行、索引与状态流转。容器可以承载代币资产、实体数据、规则实例及任意链上业务。

**区块 = 数据边界**　区块的核心职责不是哈希链主体，而是实现容器集合的数据边界——划定每批容器的提交范围，负责打包、排序与高度推进。真正的哈希关系发生在**容器与容器之间**，每个容器持有对前序容器的哈希引用，构成可验证的容器链。

**规则 = 权限**　规则不是全局共享代码，而是容器内独立生效的权限实例。同样的规则能力在不同容器之间彼此隔离——资产 A 可冻结，资产 B 不可冻结；资产 A 可增发，资产 B 总量恒定。开发者通过规则交易更新自己容器的规则，无需人工审批。

**规则实例化 = 智能合约替代**　传统智能合约将任意代码部署到链上，由虚拟机解释运行，边界模糊、难以审计。QCC 的替代路径是：规则作为系统级能力预先定义，开发者将规则实例化注入容器，规则在容器边界内独立生效，构成该容器的业务执行逻辑。

> **QCC 业务原生拒绝在链上「写代码（合约）」，倡导在链上「配规则」。**

**容器生命周期 = 进程生命周期**　容器从创建、激活、运行、冻结到注销，每个阶段均有对应的规则约束与状态记录，可被链上完整追溯。

**交易 = 系统调用（Syscall）**　一切链上行为统一表达为交易——不只是转账，还包括规则发布、资产创建、数据上链、权限变更、容器生命周期管理。交易是链上世界的**统一动作语言**，类似操作系统中的系统调用。

**子链 / 自链 = VM**　基于 QCC 原始数据，由第三方自行配置余额状态机、规则状态机、索引状态机所形成的派生系统，是运行在 QCC 之上的虚拟账本环境。QCC 是 OS，子链是 VM。

QCC 的去中心化首先体现在：**参与权开放、开发权开放、发布权开放**。

## 1.6　QCC 的统一交易观

QCC 所说的「交易」，不是狭义的转账。

在 QCC 中，凡是能够被提交到账本、形成可信记录，并引发状态变化、规则确认或权属变更的行为，都属于统一交易。转账只是其中一种。

以下行为在 QCC 中均以交易形式进入系统：

- 点对点资产转移
- 规则发布、更新与实例化
- 资产创建与定义
- 数据上链与存证
- 权限变更与容器生命周期管理
- 系统级规则调用

QCC 通过统一交易，将广义可信数据统一纳入同一个账本体系。

# 第二章　QCC 的技术基础与系统架构

## 2.1　一切链上行为统一为交易
QCC 对「交易」的定义，不是狭义的转账，而是链上世界的统一动作语言。
在传统区块链中，交易通常只代表资产从一个地址流向另一个地址；而在 QCC 中，凡是能够进入账本、形成可信记录、引发状态变化、触发规则确认或改变权属关系的行为，统一都表达为交易。
因此，在 QCC 中，以下行为本质上都属于交易：

资产转移
规则发布、更新与实例化
资产创建与参数定义
数据上链与实体存证
权限变更与容器生命周期管理
系统级规则调用
开发者、商户与业务级链上提交

QCC 之所以采用统一交易模型，是因为链上系统若要真正承载复杂业务，就不能为每一种行为发明完全不同的底层对象。QCC 选择将不同业务语义收敛为统一交易，再通过交易类型、资产域（assetId）、规则上下文、状态机处理路径、Gas 计费逻辑、签名与权限边界来区分不同交易的业务后果。
换言之，QCC 的「交易」不是单纯的支付动作，而是一次经过签名验证、支付 Gas、进入容器、参与确认并写入可信账本的标准化提交。
统一交易观带来四项核心意义：

统一验证：所有链上行为都经过同一套签名、Gas、合法性检查
统一审计：所有关键动作都进入同一账本体系
统一扩展：未来新增业务能力，无需重写底层协议范式
统一理解：开发者与节点都围绕同一基础动作模型构建系统


## 2.2　容器链结构
QCC 的核心链式结构不是传统单层区块模型，而是由容器链构成。
容器（Container）是标准的上链单元、业务承载单元、全网哈希推进单元，同时也是 QCC 的「大区块」。容器具有双重属性：

从业务抽象看，容器类似进程，是一个独立承载资产、规则、数据与状态边界的链上运行单元
从链式结构看，容器就是全网主链上的大区块，是确认、传播、落盘、同步与哈希衔接的基本对象

与之对应，区块（Block） 在 QCC 中不再是全网主链的唯一主体，而是容器内部的数据边界单元，负责划分容器内部的数据组织边界，承担排序、分段、顺序链接与局部索引职责。
QCC 因此形成双层结构：
外层：容器链

容器与容器之间通过哈希引用形成全网主链
容器承担高度推进、共识确认、全网同步与可信记录
容器是外部可感知的标准区块级对象

内层：区块边界

区块存在于容器内部
区块负责组织容器中的具体交易、数据与片段结构
区块定义数据边界，但不取代容器作为主链推进单元的地位

QCC 容器链的另一个核心组织原则：

一个容器只服务于一个 assetId 资产域。

这带来三项核心收益：资产域隔离、规则域隔离、状态域隔离——不同资产、数据业务互不混装，边界清晰，状态更新、冻结、增发、索引、审计都可围绕单一资产域独立展开。
### 2.2.1　容器的哈希构建过程
容器的最终链上哈希不是一步生成的，而是经过三个阶段逐步绑定，每个阶段都有明确的技术语义。
第一阶段：本地封装（seal）
容器在进入共识前，先在本地完成内容封装：
bodyHash    = SHA3-256(∀block: blockNumber + blockHash 拼接)
containHash = SHA3-256(containerId + bodyHash)
merkleRoot  = SHA3-256(∀block: blockHash 线性拼接)

bodyHash：对容器内所有区块的内容摘要，blockNumber 参与计算防止相同内容在不同高度产生相同哈希
containHash：将容器 ID 与内容绑定，代表「某个具体容器对某个具体内容的确认」
merkleRoot：所有区块哈希的线性摘要，用于快速索引验证

#### 实现证据：容器最终链上哈希的三阶段绑定

下面的源码摘录展示了 QCC 容器最终链上哈希并非一步生成，而是先完成本地封装，再注入共识返回的 `prevHash`、`height`、`timestamp`，最后生成正式的 `tailHash`。  
这说明 QCC 的正式链上确认单位是容器，而不是容器内部的微区块。

```java
/**
 * 最终上链：将 共识合约 返回的 PrevHash 注入，构成完整的哈希链
 */
public String overSeal(TxConsensus consensus) {
    if (consensus == null) return null;

    this.head.prevHash = consensus.prevHash;
    this.head.height   = consensus.height;
    this.head.timestamp = consensus.timestamp;

    // TailHash = Hash(ContainHash + BodyHash + PrevHash)
    String bindingString = this.head.containHash + this.head.bodyHash + this.head.prevHash;
    this.tailHash = CryptoUtil.sha3256Hex(bindingString);

    TxConsensus dataconsensus = consensus;
    dataconsensus.prevHash = this.tailHash;

    ContainerAsyncExecutor.submit(() -> {
        saveToLocal(dataconsensus);
    });

    return this.tailHash;
}
```

第二阶段：共识分配（TxConsensus）
共识层（Hub）为容器返回三个字段：
prevHash    // 前序容器的 tailHash
height      // 全网唯一高度
timestamp   // 共识时间戳
这一步解决的是：这个容器应该接到哪个前序容器之后，它在全网中的正式顺序是什么。
第三阶段：最终上链（overSeal）
注入共识返回的 prevHash 后，容器执行最终绑定：
tailHash = SHA3-256(containHash + bodyHash + prevHash)
tailHash 是容器的正式链上哈希，也是下一个容器 prevHash 的来源。这个设计确保了物理容器、逻辑内容、前序序列三者缺一不可——任何一项被篡改，都会导致整条哈希链断裂。

容器链的真正哈希关系是：tailHash(N) → prevHash(N+1) → tailHash(N+1)，链条在容器层面形成，而不是区块层面。

### 2.2.2　共识的分层定义
QCC 的共识不是单一机制，而是分层完成的：
系统共识：负责决定链的基础秩序——哪些角色拥有确认权、治理边界如何生效、质押与奖励如何约束。QCC 的系统共识由规则容器、规则治理机制以及创世纪系统规则共同定义，不是「某台服务器说了算」，而是「系统允许谁确认、按什么条件确认」这一整套规则秩序。
数据上链共识：负责解决容器如何获得快速确认并进入全网传播。当前版本采用 2/3 确认广播机制，核心目标是快速确认数据已获得足够确认条件，进入正式广播、状态同步和后续处理流程。
### 2.2.3　共识角色与演进路线
QCC V1 阶段的确认机制采用相对集中的共识角色，以保证底层结构先跑通、先稳定、先形成统一秩序。这是明确的阶段性工程选择，并不等于 QCC 对长期共识结构的最终定义。
在制度设计上，共识角色的确认权来源不是个人意志，而是系统规则——系统规则允许其确认，治理规则约束其边界，后续演进规则可以改变其产生方式与轮值方式。
V2 演进方向包括：轮流确认机制、租约式确认权、基于质押的确认权分配、规则多签代表人确认、多节点联合确认与治理约束。V2 将实现规则治理多签管理，从 V1 操作系统层去中心化，到 V2 开源与角色去中心化。
QCC 并不将「节点越多」简单等同于「越去中心化」。真正的抗女巫能力建立在：节点加入需满足链上规则条件、参与需质押或承担经济成本、确认权取决于执行权重而非简单在线。
节点开放路线：V1 阶段优先开放字节数据矿机节点，围绕实体数据上链验证基础能力；V2 阶段交易数据矿机节点逐步开源开放，引入质押参与、节点治理与状态协同。

## 2.3　状态机架构
QCC 不以「通用虚拟机解释任意代码」为核心执行路径，而以状态机架构为核心执行骨架。
基本思想是：链上写入统一通过交易完成，而交易的真实后果，则由不同状态机按明确边界执行。QCC 将系统拆解为多个职责明确的状态处理单元，每一类状态机只处理自己应该处理的问题。
余额状态机：处理资产余额、扣减、增加、冻结、解冻、分配、归属等价值流转相关状态变化。不只服务于原生币，也服务于稳定币、规则资产及未来更多金融型资产对象。
规则状态机：处理规则的发布、更新、启用、停用、实例化、权限约束与规则生效边界。决定一个容器中的资产是否可冻结、是否可增发、是否允许分红、是否允许商户回调等。
索引状态机：将链上的原始记录组织成可查询、可检索、可聚合的系统视图。QCC 若要服务真实世界业务，链上不仅要「存在」，还必须「能被检索、解释与使用」。索引状态机相当于 QCC 的链上原生检索层，类似操作系统中文件系统的索引能力，使 QCC 具备「链上数据库」的基础能力。
随着系统演进，QCC 还可以围绕共识凭证、数据路由、奖励分配、节点协同、派生子链等场景扩展更多专用状态机，但原则始终不变：一类问题由一类状态机负责，一类边界由一类状态机守住。
状态机架构的意义：可验证、可审计、可扩展、高性能、高安全——系统边界清晰，规则执行范围可控，不会隐藏在不可解释的任意代码里。

## 2.4　规则容器体系
QCC 对传统智能合约路线的替代，不是简单反对代码，而是提出了另一条路径：

不在链上部署任意业务代码，而在链上实例化规则能力。

规则不是一段面向全网自由执行的任意代码，而是被系统预定义、可被开发者调用、可在容器边界内独立生效的能力单元。规则的本质更接近系统权限、业务约束、行为边界、链上可验证的能力接口。
规则体系的核心特征：
规则能力预定义：系统先定义能力边界（发布、更新、冻结、解冻、增发、回调、奖励分配、权限更新、生命周期控制），开发者在这些边界内选择、启用、实例化，而不是绕过系统边界直接部署任意逻辑。
规则只在容器边界内生效：资产 A 启用了冻结规则，不代表资产 B 也自动可冻结；某容器启用了商户回调规则，不代表其他容器也拥有相同能力。
规则服务于业务确定性：目标不是追求「图灵完备」，而是行为明确、边界可见、后果可验证、权限可确认、执行可审计。
规则体系分层：用户级（面向开发者、商户）、资产级（面向资产发布与管理）、系统级（面向 Gas、系统地址、全网运行边界）、创世纪级（定义系统初始秩序与超级地址）。
QCC 的规则容器体系不是「弱化版合约」，而是面向真实资产、真实数据、真实业务边界设计的规则操作系统，带来更低的规则失控风险、更容易审计的系统逻辑、更可预测的 Gas 与执行边界。

## 2.5　资产与数据的统一模型
传统区块链通常把「资产」和「数据」看成两种不同世界的对象：资产负责转移价值，数据只是附属说明或外部引用。QCC 对此进行了统一重构。
在 QCC 中，资产是链上业务对象的归属域、规则域与状态域的统一标识，分为两大方向：
金融资产：原生币、稳定币、票据型资产、权益型资产及其他可流通价值对象，强调发行、流转、冻结、增发、分配、权属与清结算。
数据资产：文本、图片、订单、财报、合同、凭证、日志、应用内容以及任意字节数据，强调发布、引用、存证、版本、归属、可信来源与长期留存。
两类资产在 QCC 中采用统一结构逻辑：都有 assetId，都可通过交易进入系统，都可绑定规则，都可进入容器，都可接受状态机处理，都可形成可验证、可审计、可追溯的链上记录。
QCC 认为：数据本身也可以成为被定义、被发布、被归属、被规则化处理的资产对象。 订单、财报、合同、回执、审计线索，都可以直接进入可信账本体系，而不再只能依赖中心化数据库和截图证明。

## 2.6　基于 Netty 的字节协议
QCC 不只重构链上对象，也重构链的内部通信方式。
传统 Web 系统服务间通过 HTTP 与文本协议交互，但对于需要承载高频内部协同、容器传输、状态同步、索引更新和共识交互的链上系统，文本协议会引入过多冗余与解析成本。QCC 因此采用基于 Netty 的字节级内部通信协议。
设计原则：

面向字节而不是面向文本：内部通信优先采用字节帧结构而非 JSON/HTTP，减少传输冗余、降低解析开销、提高边界确定性
面向帧边界而不是面向页面接口：每一次内部请求都是一个明确边界、明确操作、明确载荷的字节级提交
面向统一操作语义：内部各模块调用围绕统一操作标签、统一载荷结构与统一响应模型展开，使内部网络更像系统总线
面向大载荷的帧协议边界：针对 256K 级别标准载荷及更高体量的容器化数据场景，协议层采用明确的帧边界、分段传输与重组机制

TxConsensus 的字节序列化是这一设计的直接体现——共识对象在节点间传输时，完全采用 ByteBuffer 纯字节编码，包含 nodeId、containerId、bodyHash、containHash、assetId、prevHash、height、timestamp 等全部字段，无任何文本格式开销。

节点之间不是网页互调，而是系统模块之间的字节级协作。


## 2.7　抗量子密钥与账户体系
QCC 在安全体系上从一开始就面向后量子时代——传统区块链大量依赖经典公钥密码体系，而随着量子计算能力的长期演进，经典签名体系面临潜在风险。QCC 因此在密钥体系设计上优先选择抗量子签名路线。
### 2.7.1　签名算法：ML-DSA-87
QCC 当前密钥体系采用基于格密码路线的后量子签名标准 ML-DSA-87（对应 NIST FIPS 204 标准，前称 Dilithium5），规格如下：
参数规格算法标准ML-DSA-87 / FIPS 204公钥长度2592 字节私钥长度4864 字节（标准私钥）实现库liboqs（客户端）/ BouncyCastle（服务端）跨平台统一全平台统一为 ML-DSA-87，废弃早期 Dilithium5 Java 实现

QCC 早期服务端使用 BouncyCastle 的 Dilithium5 实现，当前已全面统一为 ML-DSA-87 标准，以确保跨语言、跨平台（Java/Dart/其他第三方语言）密钥一致性。

### 2.7.2　密钥派生：助记词 + 支付密码双因素
支付密码会直接参与种子派生过程，是恢复同一密钥对不可缺少的一部分。
QCC 的密钥派生不是简单的「助记词 → 私钥」，而是助记词与支付密码共同参与的确定性派生过程：
seed = PBKDF2-SHA512(
input      = mnemonic（24个助记词 UTF-8 字节）,
salt       = "ML-DSA-87:qcc-wallet:" + paymentPassword,
iterations = 210,000 次,
keyLength  = 32 字节
)

ML-DSA-87 密钥对 = DeterministicMLDSA87.generateFromSeed(seed)
关键点：支付密码作为 PBKDF2 的 salt 参数注入，而非事后加密私钥。这意味着支付密码在密码学层面参与了种子生成过程，是密钥派生函数的输入之一。
因此，QCC 的账户恢复模型是：

助记词 × 支付密码 → 确定性种子 → ML-DSA-87 密钥对 → 地址

缺少任意一项，都无法恢复出同一把私钥和同一个地址。
### 2.7.3　助记词泄露不再等于资产失守
传统钱包模型中，助记词一旦泄露即等于账户完全暴露。QCC 的双因素派生模型改变了这一点：
即使助记词意外泄露，攻击者在缺少正确支付密码的情况下，也无法恢复出对应的 ML-DSA-87 私钥。用户仍然拥有发现风险后的应急处置窗口——紧急转移资产、重建账户、废弃旧地址。
这种保护仍依赖支付密码本身具备足够强度。如果支付密码过于简单，理论上仍可能面临暴力枚举风险。发现助记词疑似泄露时，用户应第一时间执行资产转移与账户重建。
### 2.7.4　地址格式
QCC 的普通地址（单签）由公钥按以下步骤生成：
hash32  = SHA3-256(publicKey)
hash20  = hash32[0..19]                          // 取前 20 字节
payload = 0x00 + hash20                          // version(1字节) + hash20
checksum = SHA3-256(SHA3-256(payload))[0..3]     // Double SHA3-256，取前 4 字节
address = "Q" + Base58(payload + checksum)       // 25 字节编码，Q 前缀
QCC 地址以 Q 开头，长度固定，具备内置校验和，可在客户端本地验证地址合法性。
### 2.7.5　多签地址
多签地址由多个参与方公钥与门限参数共同派生，支持 M-of-N 签名模型：
canonical   = QCCMS + ruleVersion + chainId + M + N + sorted(pubKeys)
contentHash = SHA3-256(canonical)
hash20      = RIPEMD160(contentHash)
payload     = 0x05 + hash20
checksum    = SHA3-256(SHA3-256(payload))[0..3]
address     = "M" + Base58(payload + checksum)
关键设计：

公钥按字节序排序：参与方顺序不影响地址，保证确定性
chainId 参与编码：同一组公钥在不同链上产生不同地址，防止跨链重放
版本号字段：0x05 区分多签地址与普通地址（0x00），M 开头可视
MAGIC 头 QCCMS：明确标识编码格式，为未来规则升级预留空间

多签地址以 M 开头，与普通地址 Q 开头形成视觉区分。结合规则容器，多签可进一步叠加冻结、权限变更、生命周期控制等规则，适合企业、机构与高价值资产场景。
### 2.7.6　账户体系的安全分层
QCC 的账户体系从三个层面保障安全：
层面机制保障内容链上层ML-DSA-87 抗量子签名交易合法性与权属真实性派生层PBKDF2-SHA512 双因素种子密钥不可被单因素恢复客户端层本地支付密码保护使用便利性与密钥不被滥用
QCC 选择抗量子路线，是为了回答一个更根本的问题：

如果链上数据与链上资产要承担长期可信责任，那么它们的签名体系就不能只满足今天，而必须尽量面向未来。

## 3.2.3 去中心化交易与规则资产市场

当资产能够被定义、规则能够被绑定、账本能够被统一记录时，交易市场自然就可以形成。

QCC 因此不仅可以承载稳定币与资产代币，也可以承载去中心化交易与规则资产市场。  
这里的“市场”并不只指最狭义的买卖撮合，而是指一种开放的链上流通环境：

- 资产可以被挂单
- 资产可以被撮合
- 资产可以按规则进行交易
- 交易可以进入链上确认与索引系统
- 市场行为可以被审计与追溯

QCC 在这一点上的独特性在于，它并不只是复制传统订单簿逻辑，而是提供了具有自身特色的 **Open / Win 交易模型**。

#### Open / Win 交易模型

在 QCC 中，交易发起方可以先创建一笔 `Open` 交易。  
`Open` 交易会先生成一个链上交易哈希，作为后续市场匹配与成交的基础对象。  
对手方则基于该哈希及相应条件进行匹配，并完成链上成交确认。

这种机制意味着，QCC 的市场并不是单纯“把订单挂上去”，而是让交易本身成为一个具有明确规则边界和可验证哈希对象的链上开放请求。

#### Open 交易可支持的典型形式

QCC 当前的 `Open` 交易至少可以支持以下几类典型市场行为：

- 随机分发型 Open（如红包随机空投）
- 等额分发型 Open（如等额空投）
- 交易汇率型 Open（如现货交易对象）

这意味着，QCC 的市场能力不只适用于传统意义上的买卖撮合，也适用于资产发放、规则分配与特定条件下的开放式链上分发。

#### 交易汇率对象与现货条件

在交易汇率对象（现货）场景中，`Open` 交易不仅仅是一个简单挂单，还带有明确的条件约束。  
对手方只有在满足发起方设定条件的情况下，才能完成成交。

如果一笔现货 `Open` 交易在一次匹配中无法被全部吃完，则剩余部分不会立即失效，而是可以在规定时间内继续保留在市场中。  
在当前设计下，剩余 `Open` 可在 **24 小时内持续挂单**，等待后续进一步撮合。

#### 规则资产市场的意义

QCC 在这里最独特的一点是：  
它所交易的，不只是普通代币，还可以是**带规则的资产**。

也就是说，一个进入市场的对象，可以同时带有：

- 是否允许转让
- 是否允许冻结
- 是否允许分红
- 是否允许增发
- 是否带有商户或机构限制
- 是否需要满足特定条件才能流转

这种市场，不再是“任何对象都像空气币一样自由流动”的市场，而更接近一种**规则化资产市场**。

从更市场化的角度理解，这意味着 QCC 不只是一条适合 DEX 的链，也有能力承载更接近开放民间资本市场的规则资产发行与流通体系。

换句话说，QCC 所支持的，不只是“币的交易所”，而是更广义的：

- 链上资产市场
- 规则资产市场
- 凭证流转市场
- 分红型权益市场
- 票据与业务对象的链上流通市场

这正是 QCC 在金融方向上的重要想象力之一。

# 第四章　QCC 原生币与链上规则基础

## 4.1　QCC 原生币介绍
QCC 原生币是 QCC 公链系统的基础价值单位，也是整个链上运行环境的底层燃料。
它并不是为了单纯发行一个可交易符号而存在，而是为了支撑 QCC 链上的统一交易模型、规则执行模型、数据上链模型和生态协同模型。QCC 原生币不是附属物，而是 QCC 作为区块链操作系统能够持续运行的核心基础之一。
在 QCC 的系统设计中，原生币承担以下几类角色：

作为链上交易的基础支付单位
作为规则调用与规则发布的成本载体
作为数据上链与字节发布的费用载体
作为未来质押、执行权和链上资格边界的重要基础
作为开发者、节点、商户与生态参与者之间的统一价值媒介


QCC 原生币是链上所有交易、规则、数据与生态协作的统一价值底座。


## 4.2　统一成本载体：原生币与交易模型的关系
QCC 的底层设计之一是「一切链上行为统一表达为交易」。既然如此，所有链上行为就必须有统一的成本载体——这个载体就是 QCC 原生币。
原生币不是只服务某一类动作，而是服务于整个链上系统：

为链上行为定价
为链上资源消耗计费
为系统运行提供统一燃料
为生态参与者设定统一成本边界

因此，QCC 原生币与「一切皆为交易」之间是严格闭环关系：一切行为统一为交易，一切交易需要成本，成本统一由 QCC 原生币承担。

## 4.3　总量、精度与统一数值边界
QCC 原生币总量固定为 2.1 亿枚，永不增发，不具备冻结属性。
在精度设计上，QCC 采用 9 位精度，所有金额统一以整数最小单位表示，基于 Java long 类型进行账本计算与状态处理。
这意味着：账本金额全部以整数处理，避免浮点误差进入核心状态机；总量固定，不依赖后续增发扩容；原生币本身不可冻结，作为系统基础价值单位保持明确边界。
更进一步，QCC 不仅约束原生币本身的精度，也约束全链资产体系：

QCC 链上任何代币或资产，其精度均不得超过 QCC 原生币的 9 位精度。

这为全链资产建立了统一的数值边界——无论是原生币、稳定币、规则资产还是数据型资产，最终都必须服从统一的整数精度模型与 long 类型处理规则。好处是：全链资产计算口径一致，不同资产之间的交易、结算与状态更新统一处理，状态机、索引系统与账本系统长期一致。
QCC 不追求无限精度和无限增发，而是用固定总量、统一精度和统一整数账本，建立全链稳定的价值表达基础。

## 4.4　原生币的多层角色
QCC 原生币的作用不止于支付手续费，它在整个系统中承担多个层次的角色，对不同参与者而言意义各有侧重。
对普通用户：原生币是链上支付、链上操作和链上参与的基础——每一笔转账、每一次数据上链、每一个链上动作，都以原生币承担成本。
对开发者：原生币是应用接入、规则发布、规则调用、数据发布和生态构建的底层成本单位。开发者在 QCC 上发布容器、绑定规则、定义资产，均通过原生币与系统发生关系。
对商户与机构：原生币是回调、清结算、协同账本和系统操作的统一价值媒介。商户每一次支付确认、每一次回调触发，背后都有原生币作为统一成本基础。
对节点与执行者：随着 QCC 演进，原生币还将成为质押基础、执行权边界、规则治理准入与特定市场资格约束的重要依据。确认权不是「谁架节点谁拥有」，而是通过原生币质押与规则约束共同决定。
这里最关键的一点是：

QCC 原生币不仅代表价值，也代表责任、资格与系统秩序成本。

这与只把原生币当作「手续费代币」的系统不同。在 QCC 中，原生币更像整个链上运行秩序的统一成本载体与资格边界载体——它嵌入系统内部，而不是附着在系统外部。

本章小结
第四章回答的是：QCC 原生币在系统中究竟是什么角色。
三个核心结论：

# 第五章　创世纪容器——QCC 的零高度宣言

## 5.1　创世纪容器的定义与唯一地位
测试版本创世纪容器位置： https://test.qccio.com/v1/containerHeight/0
创世纪容器，是 QCC 全链唯一的零高度容器，也是 QCC 整个系统在公网正式启动时的根起点。
它不是普通意义上的「第一个区块」，也不是一个可以后续补写的初始化文件，而是一个在协议层、节点层、状态机层同时成立的系统级原初对象。在 QCC 中，所有节点、所有状态机以及后续全部容器链的运行，都以创世纪容器的哈希和正式链 ID 为共同起点。
一旦创世纪容器以正式公网参数发布完成：它的时间戳被永久钉死，它的哈希被永久钉死，它的内容被永久钉死，它的系统地位被永久钉死。后续所有数据只能从 1 号容器继续向后写入。创世纪容器本身不能补写、不能篡改、不能替换，也不能在后续容器链运行后重新生成。
QCC 之所以把创世纪容器提升为系统级原初对象，是因为它不仅承担创世起点，还承担规则、分配、历史与文明表达的统一封印
QCC 的规则根、分配根、历史根与文明根，都从这里开始。


## 5.2　技术特殊性
创世纪容器在 QCC 中具有独立的技术地位，并不完全遵循普通容器的常规承载方式。
当前实现中，创世纪容器采用专门设计的接收与解析方式，更接近一个由系统预定义格式解析的接收文件。它可承载更丰富的文字、规则、分配和多媒体结构，并采用更方便长期保真的接收方式，而不是简单沿用普通业务容器的通用传输方式。
这种解析能力不是发布后由节点「学习」的，而是在公网启动前就已写入系统：所有节点预置了创世纪容器的原代码与解析逻辑，所有状态机都知道如何读取其中的关键结构，正式链 ID、创世纪哈希与容器格式共同构成全网启动共识。
在容量上，创世纪容器同样具备特殊设计。当前实现下，其承载体量已明显高于普通初始化对象，现阶段约为数兆级，并可根据协议需要扩展——足以同时承载创世规则、初始分配、留言、图片、语音、短视频与加密内容。

创世纪容器的特殊性，不在于「它例外」，而在于它是整个系统在启动前就约定好的唯一原初对象。


## 5.3　唯一一次的三者合一
在 QCC 的正常运行中，交易数据、规则数据与实体数据是严格隔离的——容器隔离、状态机隔离、规则边界隔离，这是 QCC 能在长期运行中保持清晰系统秩序的核心原因。
创世纪容器是唯一的例外。
在零高度时刻，QCC 通过创世纪容器完成了全链唯一的一次三者合一：

交易数据：承载创世分配、初始余额、锁仓逻辑等价值初始化结构
规则数据：承载创世级规则、系统级参数、奖励与权限边界
实体数据：承载留言、图片、语音、短视频、加密信件等可信字节内容

这种融合不是对 QCC 隔离哲学的破坏，而是更高层级的证明：

QCC 本来就能统一处理价值、规则与可信数据，只是在正常运行中，为了系统秩序，选择将它们严格隔离。创世纪容器则用唯一一次融合，把整个系统本体完整展示出来。


## 5.4　创世纪容器承载的内容结构
创世纪容器承载的，不只是一个创世哈希或初始余额列表，而是 QCC 在零高度时刻写入链上的完整原初结构。

如果借用近代政治发展的语言来类比，QCC 的治理路径更接近一种由 **创始推进 → 多签训政 → 宪政治理** 的过程。  
这意味着，QCC 作为操作系统型基础设施，最终追求的不是谁控制它，而是**让它能够脱离个人意志，依靠规则、代议与制衡长期运行。**

这正是 QCC 作为公链与操作系统的真正基础设施定位。

它至少需要 2–3 名真正理解系统底层的技术合伙人或核心协作者，共同推动协议持续演进。

所以，QCC 在创始阶段需要外部资源，不是因为它“还不存在”，而是因为：

> **它已经存在到足以值得被加速推进的阶段。**

QCC 不是一个等待被发明的项目，而是一个已经被独立开发到核心系统成型、正在进入正式主网与区域落地阶段的基础设施。  
外部资源的意义，不是决定它能否存在，而是决定它以多快速度进入更高层次的协议阶段。

QCC 的全球路径，不是“同时占领全球”，而是：

> **先建立关键区域的真实支点，再通过这些支点形成多区域协同网络。**

## 6.7　QCC 原生币与长期运行秩序

QCC 的宏观经济逻辑，最终必须回到原生币、规则与长期秩序上。

QCC 原生币并不是独立于系统外部的投机符号，而是：

- 全链交易的基础成本单位
- 规则执行与资源消耗的统一价值底座
- 质押、治理、执行权与长期秩序的重要承载对象

在前一章中已经明确，QCC 原生币采用固定总量、统一精度与整数账本模型。  
而从宏观秩序上看，它进一步承担着更深的作用：

### 第一，统一交易成本
由于 QCC 一切链上行为皆可统一为交易，因此无论是转账、规则调用、数据上链、市场参与还是未来自动化行为，最终都必须通过 QCC 原生币承担统一成本。

### 第二，统一规则秩序
QCC 的规则不是口头规则，而是链上规则。  
规则要被发布、被调用、被治理，就必须与原生币形成制度关系。

### 第三，统一激励与约束
开发者奖励、存储奖励、节点参与、治理质押、执行权边界，未来都将围绕 QCC 原生币构建。  
也就是说，QCC 原生币不仅代表价值，也代表责任、资格与秩序成本。

### 第四，统一长期运行逻辑
QCC 的目标不是靠某一家公司长期供养，而是逐步形成：

- 协议本身可运行
- 规则本身可约束
- 激励本身可分配
- 节点本身可参与
- 开发者本身可获得回报
- 治理本身可演进

从这个角度看，QCC 原生币不是系统外部的附属金融物，而是：

> **整个 QCC 长期运行秩序的价值核心。**

它把交易、规则、激励、约束、治理与全球扩展，统一到同一套价值边界之内。
